# AGENTS.md

## 执行偏好

- 执行任务时，默认先区分哪些步骤可以并行、哪些必须串行；不要把本可并行的工作保守地全按顺序执行。
- 信息收集尽量并行化，例如同时搜索相关文件、测试、调用链、配置和日志；只有存在明确依赖关系的步骤才串行推进。
- 优先走最短关键路径，先完成能推进判断和交付的最小闭环，减少过度探索和不必要的上下文铺陈。
- 验证采用分层策略：先做最小必要验证，再根据结果决定是否扩大验证范围；除非明确要求，否则不要默认执行耗时的全量检查。
- 在没有明显高风险时，可以自行做合理假设并继续执行，减少频繁确认。默认偏速度优先，但保证基本正确性。

## 项目验证

- 这是 Go 后端项目，优先使用仓库已有命令。
- 修改 Go 代码后，优先运行与改动相关的最小测试，例如 `go test ./tests/unit/...` 或更窄包路径。
- 需要扩大验证时运行 `make unit-test`，再根据风险决定是否运行 `make test`。
- 修改数据库迁移时运行 `make migrate-validate`。
- 不要默认启动依赖 Docker 的本地服务，除非任务明确需要集成验证。

## Review-Fix 循环

- 当用户要求 review 并修复代码时，先做代码 review，优先找真实 bug、回归风险、边界条件、缺失测试和高风险可维护性问题。
- Review 发现需要修改的问题后，直接修复代码，不停留在建议层面。
- 每轮修改后运行最小必要验证，再重新 review 修改结果。
- 循环上限默认为 3 轮；如果没有 P0/P1/P2 问题，或只剩风格偏好/低价值建议，则停止。
- 最终输出修改报告，包括修复内容、验证结果、剩余风险和未处理原因。
- 保持改动范围最小，不做无关重构，不回滚用户已有改动。
